## Executive summary

CE-OS is in fundamentally good health. The finance layer is the strongest part of the codebase: all aggregation funnels through one SQL helper (`_finance_kpi_window`), bucketing is Phoenix-timezone-correct on both server and client, refunds/cancellations are handled explicitly, and a forensic audit doc (`docs/FINANCE_AUDIT.md`) re-derived every number to the cent — its CRITICAL finding (passthrough sales tax in revenue) was verifiably fixed in prod the next day. Tests are real (155 passing in 1.3s, mostly behavior tests on finance/CSV/date logic), `tsc` is clean, prod security advisors are nearly clean, and RLS covers every table.

The top three issues: (1) a confirmed data-loss landmine — the admin **Revoke/Deny** action deletes the teammate's `auth.users` row, and every business table still has `user_id ... ON DELETE CASCADE`, so revoking a member silently deletes every order/expense/customer they ever created in the shared org; (2) **branch fragmentation + migration drift** — `main` is 10 commits behind the lineage that's actually live (a fresh worktree from `main`, like this review's, starts stale by default), `origin/main` is further behind still, and two finance-RPC hotfix migrations applied to prod in June exist in no git branch at all — breaking the "every number traceable to a documented query" rule at the source-control level; (3) **nothing gates deploys** — CI runs typecheck+build only (never the 155 tests), and prod ships via manual `vercel deploy --prod` from whatever the CWD holds.

The top three opportunities: surface fetch errors in `useEntity` (today every entity page renders a false "empty" state on failure), one-click security wins already queued (leaked-password protection, request-access rate limit), and a 10-minute branch consolidation (fast-forward `main` onto `unruffled-booth`, push origin) that ends the stale-worktree hazard for every future session.

## Findings

### [P0] Revoking a teammate cascade-deletes the org's business data they created
- Domain: Security / Data
- Status: CONFIRMED
- Evidence: `supabase/functions/process-access-request/index.ts:135-143` — deny/revoke calls `admin.auth.admin.deleteUser(reqRow.user_id)` (comment: "cascades to profile + any owned data"). Prod FK map (queried live): 21 tables including `orders`, `order_items`, `expenses`, `customers`, `inventory`, `cultivars`, `shipments` have `user_id REFERENCES auth.users ON DELETE CASCADE`. Approve adds users to the shared org (`index.ts:120-125`); prod has 2 members. `TODO.md:122-123` already flags the fix ("switch user_id FKs to SET NULL so deleting an account can't cascade-wipe org data") — unapplied.
- Impact: One click on Revoke for a staff member who has entered orders/expenses silently and irreversibly deletes those rows org-wide. This was correct in the single-user era; multi-tenancy made it a landmine. It also corrupts finance history (deleted expenses/orders change every KPI retroactively).
- Recommendation: The CONTRACT migration the TODO already prescribes: `user_id` FKs → `ON DELETE SET NULL` (keep `org_id` as the ownership spine; `user_id` becomes provenance). Verify Revoke afterward with a throwaway user. Until applied, treat Revoke as forbidden for members with data.
- Effort: M

### [P1] Branch fragmentation: `main` is 10 commits behind the live lineage, and 2 prod migrations exist in no git branch
- Domain: Data layer / DevOps / Finance traceability
- Status: CONFIRMED
- Evidence: Live prod serves `2e1fd06` from `claude/unruffled-booth-8e6978` (per deploy records); that branch = `main` + 10 commits (`git log HEAD..booth`: activity detail, USPS `track-shipments` fn, orders-status fixes, inventory 2-tier, sale-time COGS, plus undeployed tip `7c804a8` ad-efficiency). `origin/main` is 4 further behind local `main`. Separately, live `list_migrations` shows `finance_kpis_fix_ytd_prev_end` (20260612034938) and `finance_pnl_fix_loopvar` (20260612132821) applied to prod, and `git log --all -- "*ytd_prev_end*" "*pnl_fix_loopvar*"` is empty — those RPC bodies exist only in the database. Applied version stamps also diverge from repo filenames (repo `20260609020000_multi_tenancy_orgs.sql` vs applied `20260610003341`), so `supabase db push`-style reconciliation can't work.
- Impact: A fresh worktree from `main` — the default for every new session, including this review — starts 10 commits stale and can regress live behavior if deployed. The two DB-only hotfixes mean the repo's SQL for `finance_kpis`/`finance_pnl` is not what computes prod numbers — the finance traceability rule fails at source control even though the live math itself is sound.
- Recommendation: (1) Fast-forward `main` to `claude/unruffled-booth-8e6978` (it's a strict superset — this is merge-free) and push origin; (2) export the two hotfix function bodies from prod (`pg_get_functiondef`) into dated migration files; (3) one rule going forward: migrations are applied only from a file already committed to `main`, and deploys happen only from `main`.
- Effort: S

### [P1] Nothing gates production deploys — CI never runs the tests, and deploys bypass CI entirely
- Domain: DevOps / Testing
- Status: CONFIRMED
- Evidence: `.github/workflows/ci.yml:3-26` — triggers on push to `main` + PRs; steps are `npm run lint` (tsc) and `npm run build` only. No `npm test` (155 tests exist and pass in 1.3s) and no `npm run e2e`. Prod deploys are manual `vercel deploy --prod` from the working branch (project memory, confirmed by branch topology; `origin/main` is 4 commits behind local). The pre-commit hook is also inert on fresh clones: `lint-staged.config.js` exists but `package.json` has no `prepare` script and no `.husky/` directory is present in this worktree.
- Impact: A finance-math regression that a unit test would catch can ship to prod with zero automated checks touching it.
- Recommendation: Two-line change to `ci.yml`: add `- run: npm test` and widen the push trigger to `branches: [main, 'claude/**']`. Optionally add `"prepare": "husky"` + commit the hook, or drop the husky/lint-staged half-config.
- Effort: S

### [P1] `useEntity` swallows fetch errors — every entity page shows a false "empty" state on failure
- Domain: Data layer / UI
- Status: CONFIRMED
- Evidence: `src/hooks/useEntity.tsx:79-83` — on fetch error: `logDbError(...); setIsLoading(false); return;` — no error state is set or returned; the hook's return type has no `error` field. `StateRenderer.tsx` ships a proper `ErrorState` with retry, but useEntity-backed pages (10 files, incl. Dashboard, Shipping, Activity, Licenses, VendorDetail) can never render it. Contrast: `useFinanceOverview.tsx:98,115-116` does expose `error` and FinancesOverview renders it.
- Impact: An RLS change, expired session edge case, or Supabase outage renders as "No results" — the operator (or an agent reading the UI) concludes data doesn't exist. This directly violates the owner's own P0 trust rule in `TODO.md` ("the app must never show fake numbers as real" — an empty list that isn't empty is the same class of lie).
- Recommendation: Add `error: string | null` to useEntity's state + return, set it in the four error branches, and render `ErrorState onRetry={refresh}` in the shared list scaffolding. Pattern already exists in useFinanceOverview to copy.
- Effort: S (hook) + S (adopt in the highest-traffic 4-5 pages)

### [P1] Preview deployments share the production Supabase project
- Domain: DevOps
- Status: CONFIRMED (architecture)
- Evidence: Supabase org has exactly one CE-OS project (`jagcnaxdxiummknyrxhj` — verified via project list). `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` are baked in at build time; Vercel env vars apply to preview builds unless explicitly split per-environment. There is no staging database.
- Impact: Any preview deployment of an unmerged branch is a fully-armed client against prod data — with real auth, real RLS, and possibly schema expectations prod doesn't have (or vice versa; see the drift finding). For a solo builder this is survivable, but it's exactly how DB-ahead-of-app situations compound.
- Recommendation: Cheapest viable: in Vercel, scope the Supabase env vars to Production only so preview builds render the "Supabase not configured" screen (`SignIn.tsx:92` already handles this gracefully). A real staging project is not worth the overhead today.
- Effort: S

### [P2] Business-table RLS is role-blind: staff can read/write/delete everything, including finances
- Domain: Security
- Status: CONFIRMED (and acknowledged in code)
- Evidence: Live policy dump: every business table has one `ALL` policy `org_id IN (select private.user_org_ids())` — no role predicate. `org_memberships` policies ARE properly role-aware (owner/manager separation, owner rows protected), so the pattern exists. `AuthContext.tsx:457-460` admits it: "this is route/UI-level gating. Data-level role restriction... tracked as a follow-up." `RequireManager` hides `/finances` but a staff JWT can call `finance_kpis` or `select * from expenses` via PostgREST directly.
- Impact: Today's 2-person team makes this low-stakes; it becomes real the day a third person (or a staff-credentialed agent) joins. Staff can also bulk-delete org data (`removeMany` is org-scoped, not role-scoped).
- Recommendation: Add role checks to the sensitive subset only — `expenses`, `finance_settings`, `channel_fee_rules`, `revenue_goals`, `mileage_log` (SELECT: owner/manager) and restrict DELETE on `orders`/`customers` to owner/manager. `private.user_role_in()` already exists; this is policy-text work, not new machinery.
- Effort: M

- Impact: Invited users can choose breached passwords for an app holding the business's finances.

- Effort: S

### [P2] `request-access` has no rate limiting, and its dedupe breaks past 50 users
- Domain: Security
- Status: CONFIRMED
- Evidence: `supabase/functions/request-access/index.ts:22` (comment admits "rate-limit lightly via dedupe"); nothing throttles distinct-email spam, each of which creates a banned `auth.users` row (`index.ts:107-113`). Separately `index.ts:93-94` uses `admin.auth.admin.listUsers()` — default page size 50 — then `.find()` by email; once >50 users exist the dedupe silently misses and the flow falls through to `createUser`, which then errors on the duplicate email → user-visible 500.
- Impact: Public endpoint allows unbounded junk-user creation; dedupe has a latent correctness bug (harmless at 3 users, guaranteed at 51).
- Recommendation: Replace `listUsers()` with `admin.auth.admin.getUserByEmail(email)` (or `listUsers({ filter })`); add a simple per-IP counter row (or Turnstile) with a daily cap. Keep the anti-enumeration behavior, which is done well.
- Effort: S

### [P2] Data access is split across three competing patterns
- Domain: Architecture
- Status: CONFIRMED
- Evidence: `useEntity` (10 files), raw `supabase.from()` in pages (15+ call sites across `Settings.tsx`, `Import.tsx`, `Inventory.tsx`, `Team.tsx`, `Subscriptions.tsx`, `CultivarProfit.tsx`, `Revenue.tsx`, `Expenses.tsx`, `Customers.tsx`...), plus REST helpers (`restGet`/`rpcCall` in `src/lib/supabase.ts`) born from the SDK-hang workaround.
- Impact: Error handling, org scoping, and activity logging are guaranteed only on the useEntity path. Direct query sites must each remember `.eq("org_id", ...)` and `logActivity` — drift is how audit-trail gaps and missed error states happen. (Spot checks found org scoping present, but nothing enforces it.)
- Recommendation: Don't rewrite — converge. Rule for new code: table CRUD goes through useEntity (or a thin `useQueryOrg` for reads); RPCs through `rpcCall`. Migrate direct `.from()` call sites opportunistically when touching those pages.
- Effort: M (spread over time)

### [P2] TypeScript `strict` is off
- Domain: Type safety
- Status: CONFIRMED
- Evidence: `tsconfig.json` has no `"strict": true` (so no `strictNullChecks`/`noImplicitAny`); 136 `any`-ish casts in `src/` (0 `@ts-ignore`, which is good). Supabase types ARE generated (`src/lib/database.types.ts`) and used via `Tables<>`.
- Impact: Null-shaped bugs (the FK/session edge cases this app has already hit) are exactly what strictNullChecks catches. The `Row`/`T` mapper pattern in useEntity leans on unchecked casts.
- Recommendation: Turn on `"strict": true`, fix the fallout incrementally starting with `src/lib` (already mostly clean — tests pass in node env), then hooks, then pages. Don't chase all 136 `any`s; the deliberate SDK casts in useEntity are documented and fine.
- Effort: M

### [P2] No error reporting path — production failures vanish into the console
- Domain: Observability
- Status: CONFIRMED
- Evidence: `ErrorBoundary.tsx:17-19` and `dbErrors.ts` log to `console.error` only. No Sentry/log table/anything.
- Impact: For a solo operator, a teammate hitting an error you never hear about is the default outcome.
- Recommendation: Lightest viable: a `client_errors` table (org_id, message, stack, url, created_at, RLS insert-only) written from `ErrorBoundary.componentDidCatch` + `logDbError`, surfaced as a count in Settings→Developer. Zero new dependencies, queryable by agents. Sentry free tier is the alternative if you'd rather not own it.
- Effort: S-M

### [P2] Agent-readiness: no way to distinguish agent actions from the human's
- Domain: Agent architecture
- Status: CONFIRMED
- Evidence: `activity_log.actor_id` exists and is used everywhere (`useEntity.tsx:158-167`, importers log `actor_id: null` → rendered as "System"). But an Atlas-style agent operating with a user session would be indistinguishable from Kaden; an agent using service role writes bypass RLS and (unless it remembers) the log.
- Impact: The audit trail — the thing that makes agent operation reviewable — can't answer "did I do this or did the agent?"
- Recommendation: Minimal schema addition: `activity_log.actor_kind text default 'user'` (`user` | `agent` | 'system'), plus a convention: agents act through a dedicated `agent@` member (staff role once role-aware RLS lands) so RLS + logging apply to them like anyone else. The RPC surface (`finance_*`, `log_production_run`, `log_supply_purchase`, importers) is already a solid machine-callable API; document the idempotency keys (`external_id`, `inventory_consumed_at`) in one short doc.
- Effort: S

### [P3] Bundle: 683 kB shared chunk, duplicate animation library, dead AI dependency
- Domain: Performance
- Status: CONFIRMED
- Evidence: Build output: `index-*.js` 683 kB (202 kB gzip); recharts correctly lazy-split (the 328 kB `financeReports` chunk is recharts, loaded only on finance pages). `package.json` ships BOTH `framer-motion` and `motion` (the same library, rebranded) — all 4 importing files use `framer-motion`, so `motion` is dead weight. `@google/genai` appears in zero output chunks (receipt scan is filename-mock only per `receiptScan.ts:52-55`) — dead dependency. `qrcode` is used (QR pages).
- Impact: Slightly slower first load; dependency confusion for future work.
- Recommendation: `npm rm motion @google/genai`. Leave the 683 kB chunk alone — it's react+supabase+framer-motion+lucide for an authed internal tool; not worth manual chunking.
- Effort: S

## Product suggestions

1. **Finish the inventory/COGS rollout: enter per-unit costs, then decide the backfill.** The engine is fully live (DB + app, deployed 2026-06-30), but `inventory.cost_basis` defaults to 0 — until costs are entered per plant, `cogs_sold` and gross margin read $0 and the new Overview metrics are decorative. Then the 227 historical no-decrement sales need their standing decision: I'd recommend backfilling `order_items.cogs` only (money math) and NOT retro-decrementing stock (physical counts already reflect reality — the owner said current counts are truth). One deploy is also pending: booth tip `7c804a8` (ad-efficiency metric) is committed but not live. Effort: S (costs + deploy) / M (backfill).
2. **Uncategorized-expense sweep for tax readiness.** `finance_alerts` already surfaces uncategorized expenses; with 1,399 expense rows and the first farm-tax year closing, a dedicated "categorize everything" flow (filter: uncategorized, keyboard-driven category assignment, powered by the existing `expenseCategorization.ts` suggestions) turns tax prep from a January panic into a weekly 2-minute habit. Builds on `ExpenseCategories` + `SmartCategorizeModal`. Effort: M.
3. **Schedule F alignment pass on the category model.** The app maps expense categories to **Schedule C** lines (`scheduleC.ts`, `expense_categories` migration), but a plant nursery typically files **Schedule F** — different line set (seeds/plants, fertilizer, chemicals, supplies, custom hire). The mapping layer is already editable per-category, so this is data + one lookup table, not a rewrite — but worth deciding *before* a year of categorization hardens around C lines. Effort: S-M (pending the owner's answer below).
4. **Partial-refund support.** Order status is the only refund signal (`status='refunded'` → whole order). Etsy partial refunds will misstate revenue when they occur. Minimal version: `orders.refund_amount numeric default 0`, subtracted in `_finance_kpi_window`'s refunds CTE; Etsy sync can populate it from receipt refund lines later. Effort: M.
5. **`/scan/:code` route to activate the QR investment.** Lot labels and QR codes already encode `/scan` URLs and `qr_codes.scan_count` sits at zero (TODO P4 notes this). A single public-ish route that resolves a code → increments count → deep-links the record would make the printed labels functional on a phone in the greenhouse. Effort: S-M.
