# CE-OS — Comprehensive Audit & Expansion Report

**Date:** 2026-06-25 · **Target:** CE-OS (Canyon Exotics operations platform) · **Supabase ref:** `jagcnaxdxiummknyrxhj`
**Method:** 11 parallel domain auditors over the real code + live DB, then an adversarial verify pass on every Critical/High security & money finding. Build, typecheck, tests, and DB advisors were executed against the actual toolchain — numbers below are measured, not estimated.

---

## ⚠️ Stack correction (read first)

The audit brief described CE-OS as **"Next.js 15."** It is not. CE-OS is a **Vite 6 + React 19 + React Router 7 single-page app** on Supabase. There is no `next` dependency anywhere. The following items from the brief therefore **do not apply** and were redirected to the real stack:

| Brief asked for (Next.js) | Reality in CE-OS (Vite SPA) |
|---|---|
| Server vs. Client Components, `"use client"` | No SSR — everything is a client component; data is fetched client-side via Supabase JS |
| `next/image`, `next/font` | Raw `<img>` (audited for CLS) + Google Fonts `<link>` |
| `loading.tsx` / `error.tsx` per segment, ISR, streaming, `<Suspense>` on data | Route-level `React.lazy` + a single `<Suspense>` (already in place — 35 lazy routes) |
| Cookie auth (HttpOnly/SameSite/CSRF) | **localStorage bearer token** — so the real threat is XSS→token theft, not CSRF |
| Server actions / API routes auth | Supabase **Edge Functions** (Deno) + one Vercel cron |

Two more brief assumptions were stale: there are **155 tests (not 66)** and **no enforced 80% coverage gate** (none is configured; actual coverage is 56.7%).

## Executive summary

**Overall health: good foundations, no Critical issues, a handful of real High-severity operational and hardening gaps.** The multi-tenancy RLS model, the server-side money math, the webhook HMAC verification, and the OAuth/PKCE flow are all genuinely well-built — better than typical Supabase apps. The risks cluster in four places: an order-status UI that can corrupt inventory, the absence of HTTP security headers under a localStorage-token model, an unthrottled public sign-up endpoint, and a data-fetch hook that refetches entire tables on every realtime event.

**Findings: 114** (plus **30** customization/expansion opportunities), triaged:

| Severity | Count | Notable |
|---|---|---|
| **Critical** | **0** | — |
| **High** | **19** | order double-decrement, no CSP/headers, request-access rate-limit, recharts/vendor bundle, useEntity refetch storm, auth/money test gaps |
| Medium | 25 | manager role self-propagation, order-totals race, a11y keyboard/contrast, no manualChunks (fixed), pagination on big lists |
| Low | 49 | mostly hygiene + consistency nits |
| Nice-to-have | 21 | incl. several **verified positives** (secret confinement, service-role isolation) |

All **4 Critical/High security & money findings that went through adversarial verification were _confirmed_** (none were false positives).

**The 6 things I'd do first (highest value / risk):**
1. **`money-state#1` (High)** — Manual order-status buttons allow `shipped → pending → shipped`, which re-runs inventory consumption and silently double-decrements stock. Guard transitions DB-side or make consumption idempotent.
2. **`sec-client#1` (High)** — No CSP / security headers under localStorage-bearer auth. _(Partially fixed in this pass — 4 safe headers added; CSP provided for a preview-tested deploy.)_
3. **`sec-edge#1` (High)** — `request-access` is public with no rate limit; an attacker can mass-create banned auth users + rows and trigger an O(users) `listUsers()` scan per call.
4. **`perf-bundle#5` (High)** — `useEntity` re-runs a full `select('*')` (up to 50k rows) on every mount **and every realtime event** on an unfiltered subscription. Apply incremental row updates + a cache.
5. **`sec-rls#1` (High)** — Admin self-promotion is blocked only by an _implicit_ column grant, not a policy/trigger; one stray `GRANT UPDATE` re-opens it. Make the `is_admin` guard explicit.
6. **`test-gaps#1–5` (High)** — The auth/token layer, money formatter, and data hooks were near-0% covered. _(Partially fixed — 3 new suites added, security/money first.)_

## 1. Security & Correctness Review


### Security: RLS &amp; Authorization (Supabase row-level security + app-level authz)

_The multi-tenancy RLS model is, overall, well-engineered and notably better than typical Supabase apps: every one of the 36 public tables has RLS enabled, there are ZERO `USING(true)`/`WITH CHECK(true)` policies, every data table carries BOTH a matching `USING` and `WITH CHECK` clause (no read/write asymmetry), and cross-org writes are defended in depth by (a) a `private.enforce_row_org()` BEFORE-trigger present on all 31 data tables that stamps/validates `org_id` server-side and (b) WITH CHECK clauses gated on `org_id IN (select private.user_org_ids())`. The client (`useEntity.tsx`) strips `id`/`user_id`/`org_id` on UPDATE, and the org-membership policies correctly block managers from creating/becoming owners while a `private.enforce_last_owner()` trigger prevents orphaning an org. The privilege-escalation surfaces that exist on paper (`profiles.is_admin`, `org_memberships.role`) are largely neutralized: `authenticated` lacks UPDATE on the `profiles.is_admin` column AND `profiles` has no DELETE policy, so the obvious self-promote-to-admin paths are blocked. The real residual risks are: (1) the `profiles self update` policy and is_admin protection rely entirely on an invisible Supabase column-grant default rather than an explicit policy/trigger guard — a single stray `GRANT UPDATE ON profiles` re-opens admin self-promotion; (2) `org_memberships.role` IS UPDATE-grantable to authenticated, letting a manager self-/peer-promote staff→manager (owner is blocked, so blast radius is limited but the manager tier gains finance access); (3) `integration_config` correctly has zero policies + revoked grants (service-role-only — the right design for the Shopify token); (4) co-member `profiles` SELECT leaks `is_admin` to all teammates (minor); (5) several `UPDATE` policies have a `USING` but null `WITH CHECK`. App-level route guards (`RequireManager`/`RequireAdmin`) are correctly applied to all sensitive routes; one mobile nav link is mis-gated but harmlessly (server RLS still blocks)._


**[F1] (High · effort S · conf high · **verify: confirmed** (→Medium)) profiles admin self-promotion blocked ONLY by an implicit column-grant, not by policy/trigger — fragile privilege-escalation guard**  

- **Where:** `supabase/schema.sql:33 ("profiles self update" policy); live grant: information_schema.column_privileges profiles.is_admin (authenticated = SELECT,INSERT,REFERENCES; no UPDATE)`

- **Evidence:** The `profiles self update` policy is `USING (auth.uid() = id) WITH CHECK (auth.uid() = id)` with NO column restriction and NO trigger guarding `is_admin` (only `set_updated_at` exists on profiles). `is_admin` is the sole gate for the `admin_emails`/`access_requests` policies and the access-request approve/deny edge function (process-access-request/index.ts:58-61), i.e. it controls who can let new users into the system. Self-promotion (`UPDATE profiles SET is_admin=true WHERE id=auth.uid()`) is currently blocked ONLY because the `authenticated` role lacks the UPDATE privilege on the `is_admin` column (verified: authenticated has SELECT/INSERT/REFERENCES on is_admin, not UPDATE) and because `profiles` has no DELETE policy (so delete-then-reinsert-with-is_admin=true is also blocked). This protection is invisible in the migrations — it is a Supabase default column-grant — so any future `GRANT UPDATE ON public.profiles TO authenticated` (common when adding a new self-editable column) silently re-opens full admin self-promotion to every authenticated user.

- **Fix:** Make the guard explicit and defense-in-depth: add a BEFORE UPDATE trigger on `profiles` that raises if `new.is_admin IS DISTINCT FROM old.is_admin` when `auth.uid() IS NOT NULL` (mirroring `enforce_row_org`'s trusted-context bypass), OR split the policy so the self-update WITH CHECK asserts `is_admin = (select is_admin from profiles where id = auth.uid())`. Do not rely on column-level grants as the sole control. Keep is_admin mutation to the SECURITY DEFINER `handle_new_user`/service-role only.


**[F2] (Medium · effort S · conf high) Manager can promote staff (or self) to 'manager' via org_memberships.role UPDATE, gaining finance access**  

- **Where:** `20260609020000_multi_tenancy_orgs.sql:297-307 ("memberships manage update"); live grant: column_privileges org_memberships.role (authenticated = UPDATE)`

- **Evidence:** `authenticated` holds UPDATE on the `org_memberships.role` column, and the `memberships manage update` policy allows the update when the actor is owner OR (manager AND target row role <> 'owner'). The WITH CHECK only forbids setting role='owner'; it does NOT forbid a manager from setting role='manager'. So a manager can elevate any staff member — or, since managers can touch their own non-owner row, keep peers at manager — to the 'manager' tier. The 'manager' tier is exactly what unlocks all role-gated finance/sensitive tables (expenses, vendors, supplies, finance_settings, revenue_goals, production_*, etc., per their `user_role_in(org_id) IN ('owner','manager')` policies). The owner tier remains protected (WITH CHECK blocks role='owner', and enforce_last_owner protects demotion), so blast radius is bounded below owner.

- **Fix:** If managers should not be able to expand the manager pool, tighten the policy so only owners can grant the 'manager' role (e.g. WITH CHECK: `user_role_in(org_id)='owner' OR (user_role_in(org_id)='manager' AND role='staff')`). At minimum document that 'manager' is self-propagating among managers. Consider revoking column UPDATE on `role` from authenticated and routing role changes through a SECURITY DEFINER RPC that enforces the intended matrix.


**[F3] (Nice-to-have · effort S · conf high) integration_config is correctly service-role-only (RLS on, 0 policies, grants revoked) — Shopify webhook token not reachable by anon/authenticated**  

- **Where:** `20260611150000_integration_config.sql:5-13; live: pg_class integration_config rls_enabled=true policy_count=0; role_table_grants shows only postgres/service_role`

- **Evidence:** RLS is enabled with zero policies, and `revoke all on public.integration_config from anon, authenticated` is applied — live grants confirm only `postgres` and `service_role` have any privileges. With RLS on and no permissive policy, anon/authenticated reads/writes are denied even if a grant were accidentally present. This is the correct pattern for holding `shopify_webhook_token`: the token is consumed by the Deno edge function via the service role (which bypasses RLS), and no client path can read it. This is a positive note, not a defect — confirming the design is intentional and sound.

- **Fix:** No action required. Optionally add `alter table integration_config force row level security` for belt-and-suspenders against a future table owner mishap, and add a regression test asserting an authenticated client gets 0 rows from integration_config.


**[F4] (Low · effort M · conf high) Co-member profiles SELECT policy exposes is_admin (and full profile row) to every teammate in the org**  

- **Where:** `20260609020000_multi_tenancy_orgs.sql:364-372 ("profiles co-member read"); live pg_policies profiles`

- **Evidence:** The `profiles co-member read` policy grants SELECT on the WHOLE profiles row (no column restriction — PostgREST can request `select=is_admin`) to any user whose id is in a shared org. Intended use is reading teammates' `display_name` for the Team page, but it also discloses each member's `is_admin` flag and `updated_at` to all co-members. Low impact (same-org only, no secrets in the table), but it leaks the admin roster to all staff.

- **Fix:** If is_admin disclosure is undesirable, expose teammate display data through a view or RPC that selects only `id, display_name`, or move display names to a non-sensitive table. Otherwise accept and document.


**[F5] (Low · effort S · conf medium) Several UPDATE policies have USING but null WITH CHECK (write-side not re-validated post-update)**  

- **Where:** `live pg_policies: access_requests "access_requests admin update" (with_check=null); profiles "profiles self update" has WITH CHECK but admin_emails/access_requests admin policies rely on USING only`

- **Evidence:** `access_requests admin update` has `qual` (USING) = admin EXISTS check but `with_check = null`. With a null WITH CHECK, Postgres falls back to the USING expression for the new row, so in this specific case the post-image is still constrained to admins — but the pattern (USING without an explicit WITH CHECK) is a latent footgun: if the USING predicate ever referenced the OLD row state in a way that the NEW row could violate, the write would not be re-checked. Here the predicate is row-independent (admin check), so there is no current hole, but it is the kind of asymmetry the audit asked to flag.

- **Fix:** Add explicit `WITH CHECK` clauses mirroring the USING predicate on all UPDATE policies for clarity and future-proofing (access_requests admin update, admin_emails admin policies). No behavior change today; prevents accidental write holes if predicates evolve.


**[F6] (Low · effort S · conf high) Mobile 'More' sheet links /import to ALL members while desktop nav gates it to managers (cosmetic — route + RLS still enforce)**  

- **Where:** `src/components/Layout.tsx:428 (mobile /import link, no canManage guard) vs Layout.tsx:248-261 (desktop guarded by canManage); App.tsx:92 (/import wrapped in RequireManager)`

- **Evidence:** On desktop the Finances, Licenses, and Import nav links are gated by `canManage` (orgRole owner/manager). In the mobile 'More' sheet, `/import` (Layout.tsx:428) is rendered for every member with no `canManage` guard (Team/Access are correctly `isAdmin`-gated at 429-430). A staff user tapping it hits `RequireManager` (App.tsx:92), which redirects to `/`, and the underlying import writes to supplies/expenses/etc. that are RLS-restricted to owner/manager anyway. So this is a UI inconsistency, not an access-control bug.

- **Fix:** Wrap the mobile `/import` link (and add Finances/Licenses for parity) in `{canManage && ...}` to match the desktop nav and avoid a confusing redirect for staff.


**[F7] (Low · effort S · conf medium) access_requests anon-insert policy defined in schema.sql is NOT present in production (request flow moved to edge function) — verify intended**  

- **Where:** `supabase/schema.sql:104-110 + 120 ("access_requests anon insert" policy + grant) vs live pg_policies (only 2 policies: admin select, admin update)`

- **Evidence:** schema.sql defines an `access_requests anon insert` policy plus `grant insert ... to anon`, but the live DB shows only `access_requests admin select` and `access_requests admin update` (policy_count=2). The public sign-up/request path now goes through the `request-access` edge function (verify_jwt=false, service-role), so the direct anon-insert policy is no longer needed and inserts by anon are denied (RLS on, no INSERT policy). This is a safe state (no anon write path), but the migration source (schema.sql) is now out of sync with prod, which can mislead future maintainers into re-adding the anon-insert grant.

- **Fix:** Reconcile schema.sql with the deployed state: remove or comment the obsolete `access_requests anon insert` policy/grant, and note that access requests are inserted exclusively via the service-role `request-access` edge function.


**[F8] (Low · effort S · conf high) RLS auth.<fn>() re-evaluated per-row on profiles/admin_emails/access_requests/activity_log (auth_initplan) — perf, and a reminder these are public-role policies**  

- **Where:** `supabase/schema.sql:31,33,49-57,111-118; live pg_policies profiles/admin_emails/access_requests (roles={public}, bare auth.uid())`

- **Evidence:** Per the live policy dump and DB advisor, profiles/admin_emails/access_requests policies call `auth.uid()` (and EXISTS subqueries) un-wrapped, so Postgres re-evaluates them per row instead of once. The data-table org policies already use the wrapped `(select private.user_org_ids())` form (good), but these legacy public-role policies do not. Also note these policies target the `public` role (not `authenticated`), which is correct for the self-profile bootstrap but means they also apply to `anon` — harmless here because `auth.uid()` is null for anon, yielding no rows.

- **Fix:** Wrap as `(select auth.uid())` in the profiles/admin_emails/access_requests policies to satisfy the auth_initplan advisor and improve scan performance on these tables. No security change. (Tracked separately under the performance section; logged here for completeness since these are authz policies.)


**[F9] (Nice-to-have · effort L · conf high) Generic (non-finance) tables grant full read/write to ALL org members including staff — orders/order_items/customers/subscriptions are not role-gated**  

- **Where:** `20260609020000_multi_tenancy_orgs.sql:208-254 (generic_tables); live pg_policies orders/order_items/customers/subscriptions "org access" (no user_role_in clause)`

- **Evidence:** By design, tables in `generic_tables` (orders, order_items, customers, subscriptions, listings, shipments, inventory, tasks, etc.) are accessible (ALL) to any org member, with no `user_role_in` restriction — only finance/sensitive tables add the owner/manager check. This means a 'staff' member can read and write customer PII, order totals, and subscription records. This appears intentional (staff operate orders/shipping), and AuthContext.tsx:457-460 explicitly notes data-level role restriction for non-finance reads is a tracked follow-up. Flagging so the product owner can confirm staff SHOULD see customer/order/revenue-adjacent data.

**[F10] (Nice-to-have · effort S · conf high) process-access-request edge function correctly re-verifies admin via caller JWT before service-role actions**  

- **Where:** `supabase/functions/process-access-request/index.ts:44-61, 72-74`

- **Evidence:** The function reads the caller's JWT, calls `userClient.auth.getUser()` to authenticate, then checks `profiles.is_admin` THROUGH the user-scoped client (subject to RLS) before instantiating the service-role admin client. Approve/deny/revoke privileged operations (unban, deleteUser, membership upsert) only run after the 403 admin gate. Input is validated (action allow-list, request_id required) and denial_reason is length-capped. This is the correct trust boundary for a privileged edge function and a positive note.

### Security — Edge Functions & Serverless Endpoints

_Six serverless entry points were mapped: five Supabase Edge Functions (shopify-webhook, etsy-oauth, etsy-sync, request-access, process-access-request) and one Vercel cron (api/cron/subscriptions.ts). Overall the auth design is thoughtful and several things are done RIGHT: shopify-webhook uses an HMAC SHA-256 verify with a constant-time compare and a timing-safe token fallback; etsy-oauth implements full OAuth2 + PKCE (S256) with a CSRF `state` param and clears one-time secrets after use; request-access is deliberately public but returns a generic response to prevent email enumeration and caps input lengths; process-access-request is verify_jwt=true at the gateway AND re-checks profiles.is_admin in code (defense-in-depth); the Vercel cron fails CLOSED if CRON_SECRET is unset. I verified live that integration_config (which holds shopify_webhook_token + Etsy secrets) is granted only to postgres/service_role with zero policies — anon/authenticated have NO read path, so the advisor's 'RLS with zero policies' is intentional and safe. I also verified process_due_subscriptions EXECUTE is granted only to service_role, so the billing RPC cannot be triggered by an anon/authenticated PostgREST call. The real residual risks are: NO rate limiting anywhere (request-access lets an anon create banned auth.users + DB rows unboundedly, and calls admin.auth.admin.listUsers() on every request — a cost/DoS + table-bloat vector); the shared URL-token auth on etsy-sync/etsy-oauth is passed as a query string (logged in plaintext in edge/access logs and the cron job definition); the HMAC verify spreads the MAC bytes via String.fromCharCode(...) which can blow the call-stack on hostile input (minor); a few endpoints leak internal error detail; and CORS is wildcard on the public functions. None of these are remote-code or data-exfil class given the service-role key stays server-side, but the request-access abuse surface is the standout._


**[F11] (High · effort M · conf high · **verify: confirmed** (→Medium)) request-access has NO rate limiting — anon can mass-create banned auth users + access_request rows (resource exhaustion / table bloat)**  

- **Where:** `supabase/functions/request-access/index.ts:52-133`

- **Evidence:** verify_jwt=false (confirmed live) and the function is fully public. Each accepted POST with a unique valid email + 8-char password calls admin.auth.admin.createUser(...) (line 107) and inserts an access_requests row (line 119). The only throttle is per-email dedupe (lines 77-98) — a script iterating unique emails bypasses it entirely. There is no IP/global rate limit, no CAPTCHA, no max-pending cap. Each request also unconditionally calls admin.auth.admin.listUsers() (line 93), an expensive full-user enumeration, so the cost grows with every call AND with directory size.

- **Fix:** Add a global/IP rate limit (e.g. a counter in integration_config or a Postgres rate-limit table keyed by IP + minute window, or front it with a hCaptcha/Turnstile token verified server-side). Cap total outstanding 'pending' access_requests. Replace the per-call listUsers() scan with a targeted lookup (query access_requests / a dedicated index) so cost is O(1), not O(users).


**[F12] (Medium · effort M · conf high) Shared URL token (?token=) passed as a query parameter on etsy-sync / etsy-oauth — logged in plaintext**  

- **Where:** `supabase/functions/etsy-sync/index.ts:462; supabase/functions/etsy-oauth/index.ts:88; supabase/migrations/20260611160000_etsy_sync_cron.sql:26-27`

- **Evidence:** etsy-sync reads `new URL(req.url).searchParams.get('token')` and etsy-oauth reads `url.searchParams.get('token')`. The pg_cron job builds the URL with `...?token=' || (select value from integration_config where key='etsy_sync_token')` (migration line 26-27), so the long-lived sync token is embedded in the cron.job definition AND appears in every Supabase edge-function access log, pg_net request log, and any proxy in between. Query-string secrets are a well-known leak vector (logs, Referer, browser history for the OAuth GET).

- **Fix:** Move the token to a request header (e.g. X-Sync-Token) for etsy-sync and read it from the header in pg_net's `headers` jsonb instead of the URL. For etsy-oauth the browser GET is harder to avoid, but rotate etsy_sync_token after the one-time authorization and scope log retention. At minimum, document that the token is log-exposed and must be rotated if logs are shared.


**[F13] (Low · effort S · conf medium) HMAC verification spreads MAC bytes via String.fromCharCode(...new Uint8Array) — fragile, and btoa over raw bytes is non-standard**  

- **Where:** `supabase/functions/shopify-webhook/index.ts:52`

- **Evidence:** `const digest = btoa(String.fromCharCode(...new Uint8Array(mac)));` — the spread of a Uint8Array into String.fromCharCode pushes every byte as a function argument. For the 32-byte SHA-256 MAC it is fine, but this is a brittle pattern (large arrays can exceed the JS argument/stack limit) and relies on btoa over a binary string, which only works because each byte is <256. Functionally correct for Shopify's fixed-length signature, hence Low.

- **Fix:** Use a byte-safe base64 encoder (loop or a helper) rather than spreading into String.fromCharCode, and compare the computed digest against the header consistently. Optional hardening only.


**[F14] (Medium · effort M · conf high) shopify-webhook accepts ANY body when SHOPIFY_WEBHOOK_SECRET is unset and relies solely on the URL token — no replay/body-size protection**  

- **Where:** `supabase/functions/shopify-webhook/index.ts:26,42-43,66-78`

- **Evidence:** WEBHOOK_SECRET defaults to '' (line 26). verifyHmac returns false immediately when the secret is empty (line 43), so if SHOPIFY_WEBHOOK_SECRET is not configured the ONLY auth is the `?token=` fallback (lines 69-77) — a static bearer in the URL with no HMAC, no timestamp/nonce, and therefore no replay protection. `await req.text()` (line 59) reads the full body with no size cap before parsing, and topic is taken from a spoofable client header `x-shopify-topic` (line 80) used only to skip non-create topics. An attacker holding the URL token can replay or inject synthetic orders/customers via importNormalizedOrder (writes orders, customers, shipments, activity_log under service role).

- **Fix:** Treat HMAC as mandatory when the integration is live (reject token-only path once SHOPIFY_WEBHOOK_SECRET is set, or require it). Add a max body-size guard before req.text()/JSON.parse. Consider a dedupe/replay window keyed on payload.id (importNormalizedOrder already dedupes by external_id, which mitigates replay-as-duplicate, but a forged unique id still inserts a row). Don't trust x-shopify-topic for anything security-relevant.


**[F15] (Low · effort S · conf high) Error responses leak internal exception detail to token-gated callers**  

- **Where:** `supabase/functions/etsy-sync/index.ts:502; supabase/functions/process-access-request/index.ts:92,141`

- **Evidence:** etsy-sync inspect path returns `detail: err instanceof Error ? err.message : String(err)` in the HTTP body (line 502). process-access-request returns `detail: unbanErr.message` (line 92) and `detail: delErr.message` (line 141) to the caller. These surface raw Supabase/Etsy error strings (which can include internal identifiers, constraint names, or upstream API messages) to whoever holds the sync token / is an admin. Lower severity because the inspect path is token-gated and the access-request path is admin-gated, but it still violates 'error messages don't leak internals'.

- **Fix:** Return a generic user-facing message in the HTTP body and log the detailed error server-side (console.error already does this). Drop the `detail` field from responses, or gate it behind a debug flag.


**[F16] (Low · effort S · conf medium) Wildcard CORS (Access-Control-Allow-Origin: *) on public edge functions**  

- **Where:** `supabase/functions/request-access/index.ts:30; supabase/functions/process-access-request/index.ts:24`

- **Evidence:** Both functions set `'Access-Control-Allow-Origin': '*'`. For request-access this lets any origin POST (acceptable since it is intentionally public, but combined with no rate limit it widens the abuse surface to any web page). For process-access-request, the wildcard means any origin can attempt the call from a victim admin's browser session; it is still protected by the bearer-token requirement (Authorization header, not a cookie), so this is not a classic CSRF, but tightening the origin is good hygiene.

- **Fix:** Restrict Access-Control-Allow-Origin to the app's known origin(s) rather than '*' for these functions, since both are first-party app calls, not third-party webhook targets.


**[F17] (Low · effort S · conf medium) process_due_subscriptions / log_subscription_charge can be invoked by the cron path but financial worker has no idempotency guard against concurrent runs beyond date advance**  

- **Where:** `api/cron/subscriptions.ts:32-37; supabase/migrations/20260611270000_subscriptions_selfmaintaining.sql:64-89`

- **Evidence:** The Vercel cron is correctly protected (CRON_SECRET, fail-closed, lines 18-23) and process_due_subscriptions EXECUTE is granted only to service_role (verified live: acl 'postgres=X/postgres, service_role=X/postgres') so anon/authenticated cannot trigger billing via PostgREST — both good. However process_due_subscriptions loops and calls log_subscription_charge which inserts an expense then advances next_renewal in two statements without a row lock (no SELECT ... FOR UPDATE). Two overlapping invocations (e.g. a manual curl racing the scheduled run) could both read next_renewal <= today and double-insert a charge for the same subscription before either advances the date. The daily schedule makes this unlikely but the worker is not concurrency-safe.

- **Fix:** Add `for update skip locked` when selecting due subscriptions in process_due_subscriptions, or make log_subscription_charge advance the date conditionally (UPDATE ... WHERE next_renewal = v_base) and only insert the expense if the UPDATE affected a row. This guarantees exactly-once even under concurrent invocation.

### Security/Correctness — Money handling, Order state machine, Inventory ledger

_Money is stored correctly as Postgres numeric(10,2) (decimal dollars, exact) for orders.subtotal/shipping/tax/total, order_items.price, expenses.amount, and supplies.cost — confirmed via live DB introspection. The authoritative finance math (_finance_kpi_window, finance_cashflow, log_supply_purchase, log_production_run) all runs server-side in exact numeric and is the single source of truth; src/lib/finance.ts explicitly documents itself as a client-side mirror for previews only, with roundMoney/roundCost boundary rounding, and is well unit-tested (finance.test.ts). This is a genuinely strong design: the COGS RPCs are atomic SECURITY INVOKER (one event → one entry → many rollups), webhook order imports are idempotent by external_id and forward-only on status, and consume_inventory_for_order fires exactly once on the first transition into shipped/delivered, clamps stock at zero (least()), and logs shortfalls rather than hiding them — so duplicate webhooks cannot double-decrement. The real gaps are: (1) the manual Orders UI exposes every status as a free button with no transition guard, so shipped→pending→shipped re-runs inventory consumption and decrements stock a second time; (2) several multi-step writes are done as sequential client-side calls (createOrder, recalcTotals, the weather sweep) with no transaction boundary and a stale-closure read-modify-write that can lose concurrent edits; (3) numeric columns are typed `number` in database.types.ts, so every client read coerces exact DB numerics into float64 — harmless where the server total is displayed, but the client re-summations in CultivarProfit/Dashboard/NetProfitWaterfall accumulate float and can fail to reconcile to the server figure by a penny. Money stored in unit-cost columns (unit_cost_snapshot, total_cost, labor_rate, supplies.on_hand qty) is unconstrained numeric (no scale), which is exact in PG but means moving-average unit costs carry full precision._

_CE-OS is a dark-themed (#0E0F11) Vite/React 19 SPA. The team has done meaningful a11y groundwork: the shared Input/SignIn forms use proper `<label htmlFor>` association; most icon-only buttons in chrome (Layout topbar, Modal close, Toasts dismiss, DataTable pager, PhotoUploader delete) carry `aria-label`; the Toggle is a real `role="switch"` with `aria-checked` and a `focus-visible` ring; all `<img>` tags have alt text; toasts use `role="status"` + `aria-live`; index.html sets `lang="en"`; and a global `:focus-visible { outline: 2px solid }` rule gives visible focus by default. However there are systemic gaps that block AA: (1) no focus trap, initial-focus, or return-focus in ANY dialog (shared Modal, Dialog, CommandPalette, KeyboardReference, TasksPanel, NotificationCenter) — keyboard/AT users can tab out behind the modal; (2) clickable `<div>`/`<Card>`/`<tr>` rows (Inventory cards, DataTable rows, CommandPalette results, NotificationCenter items) are mouse-only — no `tabIndex`/`role`/`onKeyDown`, failing keyboard operability; (3) `text-tertiary` (#5C6066) fails 4.5:1 normal-text contrast on every background (3.03:1 on base, 2.78:1 on elevated, 2.52:1 on active) and is used widely for hints/labels/metadata; (4) no skip-link to bypass the 12+ item nav; (5) no `prefers-reduced-motion` handling anywhere (framer-motion modals, slide-in toasts, animate-pulse skeletons, animate-spin); (6) filter/form `<select>`s across ~20 sites and all search inputs rely on placeholder only with no `<label>`/`aria-label`; (7) no `<th scope="col">` on any table; (8) NavLinks don't expose `aria-current`; (9) recharts charts have no text alternative/`accessibilityLayer`. Severity is concentrated in keyboard access and contrast._

### UI — Visual consistency, polish, responsive layout, and loading/empty/error states

_CE-OS has a genuinely strong, cohesive visual foundation: a well-defined dark design-token system in src/index.css (bg/border/text/status/accent CSS vars), a shared primitive set (Button, Badge, Card, Input, StatTile, Modal, StateRenderer, DataTable, Toasts), tabular-nums on all money, thoughtful mobile affordances (bottom nav, safe-area utils, iOS 16px-input zoom guard, dvh sizing), and the finance section in particular (FinancesOverview, Goals, Revenue) is polished with real loading skeletons AND real error states wired to its hooks. The dominant systemic issue is that error states are effectively dead app-wide: useEntity (src/hooks/useEntity.tsx) swallows every fetch error and returns empty data, and useDataState (src/hooks/useDataState.tsx) derives isError purely from a developer toggle — so every CRUD/list page (Orders, Customers, Inventory, Cultivars, Vendors, Supplies, etc.) silently shows a "nothing here yet" empty state on a real network/RLS failure. ErrorState is imported by only 2 pages (Inventory, Goals) and Inventory's can never actually fire. Secondary themes: modal patterns are fragmented (shared Modal vs Dialog vs ~6 hand-rolled fixed-overlay modals, none of the 5 slide-out detail panels close on Escape, 17 native confirm() dialogs), ~39 raw &lt;select&gt; elements drift across at least 4 padding/bg/focus variants because there is no shared Select primitive, border-radius and page-title typography drift (rounded-md/lg/xl/[8px] mixed; 3 pages use text-xl titles + p-md while 32 use text-2xl + p-8), the Suspense route fallback is a blank screen, and a couple of pages (Capacity, CultivarProfit) have no loading state so they flash an empty state before data. None of these are functional blockers, but together they read as polish/consistency debt concentrated outside the flagship finance views._


**[F57] (High · effort L · conf high) Fetch errors are invisible across all useEntity pages — users see a misleading empty state on failure**  

- **Where:** `src/hooks/useEntity.tsx:79-83 (error path) and :44-54 (return type has no isError); consumers e.g. src/pages/Orders.tsx:222, src/pages/Customers.tsx:215`

- **Evidence:** fetchAll() on error calls logDbError + setIsLoading(false) and returns, leaving data=[] and exposing no error flag. Every consumer computes isEmpty = !isLoading && data.length === 0 and renders EmptyState. So a network drop / RLS denial / 500 on Orders renders 'No orders yet — Create an order to start tracking' instead of an error with retry.

- **Fix:** Add an `error` field to useEntity's state and return it; in each list page branch loading → error (ErrorState with onRetry={refresh}) → empty → data, mirroring what Goals/FinancesOverview already do via their hooks.


**[F58] (High · effort M · conf high) useDataState.isError is wired to a dev toggle, not real failures — Inventory's ErrorState can never fire in production**  

- **Where:** `src/hooks/useDataState.tsx:25 (isError: settings.errorMode); src/pages/Inventory.tsx:99,337-338`

- **Fix:** Drive isLoading/isError from the real data source (useEntity), not a timer/toggle. Keep the dev toggles as an override layered on top of real state.

## 3. User Customization


### User Customization

_CE-OS today offers a thin customization surface: a single-toggle "compact density" (localStorage via usePersistedState, ceos: prefix), editable expense categories (finance_settings.expense_categories jsonb, owner/manager-gated), channel-fee rules, a notification-prefs panel that persists to profiles.notification_prefs but is explicitly "Preview — wiring in progress" with no delivery, a hardcoded ⌘K command palette + static keyboard-shortcut reference, and a Dashboard operations/reporting toggle that isn't even persisted. There is NO theme/dark-light/accent control (the app is hardcoded dark via @theme in src/index.css:4-28; only `font-compact` is conditionally applied at Layout.tsx:166), NO saved views / column config / configurable dashboard, NO per-user preferences table (finance_settings is the only settings table; it's org-level and owner/manager-gated), and NO message templates. A notable half-built seam: finance_settings already has org-level `timezone` and `home_state` columns, yet src/lib/dates.ts hardcodes BUSINESS_TZ = "America/Phoenix" (dates.ts:17) and src/lib/format.ts uses the browser locale with no override — so the DB-backed timezone is never read. Statuses (src/lib/status.ts), propagation stages (Capacity.tsx:13), nav items, and command-palette commands are all hardcoded arrays. Persistence is split: ephemeral UI state lives in localStorage (usePersistedState), and the only durable per-account store is profiles (display_name, notification_prefs). The single highest-leverage move is to add a per-user `user_preferences` table (+ org_settings for shared config) and route theme, density, default filters, saved views, and dashboard layout through it so settings survive across devices and the prod Supabase login. Below: 16 opportunities spanning theming, dashboards/saved views, defaults, notifications, custom fields/statuses, workflow/templates, shortcuts/quick-actions, and the persistence layer._


**[O80] (high impact · effort M · persistence) Add a per-user user_preferences table as the durable home for all personal settings**  

- Today the only durable per-account store is `profiles` (display_name, notification_prefs jsonb). Everything else personal — settings.density, notifications array, Capacity config — lives in localStorage under the `ceos:` prefix (usePersistedState.tsx:3) and therefore does NOT survive a device switch, a browser-cache clear, or the 'Reset local data' dev button (Settings.tsx:483-491). finance_settings is org-level AND owner/manager-gated, so it can't hold personal prefs for non-managers. A dedicated per-user table is the foundation every other customization opportunity below depends on.

- **Approach:** Create one row-per-user JSONB-bag table so the schema never needs migration as new prefs are added. Schema sketch:

create table public.user_preferences (
  user_id uuid primary key references auth.users(id) on delete cascade,
  org_id uuid references organizations(id) on delete cascade,  -- prefs are usually global, but org-scoping lets a future multi-org user differ per org
  prefs jsonb not null default '{}'::jsonb,  -- {theme, density, accent, defaultFilters, dateFormat, savedViews, dashboardLayout, quickActions...}
  updated_at timestamptz not null default now()
);
alter table public.user_preferences enable row level security;
create policy "own prefs" on public.user_preferences
  for all using ((select auth.uid()) = user_id) with check ((select auth.uid()) = user_id);

Note the (select auth.uid()) wrapping — the live advisors flagged auth_initplan re-eval on profiles/admin_emails; do it right from row one. Mirror the existing ExpenseCategoriesContext pattern (read on mount, optimistic local state, maybeSingle upsert). Provide a usePreferences() context + a usePref<T>(key, default) hook that reads/writes a single JSON path and debounces the write. Migrate usePersistedState callers (AppContext settings/notifications, Capacity config) to read localStorage as a fallback cache but treat the table as source of truth.


**[O81] (medium impact · effort M · persistence) Add an org_settings table for shared, non-finance org configuration**  

- finance_settings is the only org-scoped settings table and it is gated to owner/manager AND semantically finance (labor rate, mileage, expense categories). There's no neutral place to store org-wide, non-finance preferences that every member should see consistently — e.g. brand accent within palette, default low-stock threshold (currently the magic constant LOW_STOCK_THRESHOLD=10 in Dashboard.tsx:23), business display name, default order-list sort, message templates. Overloading finance_settings would wrongly hide these behind the manager gate.

- **Approach:** create table public.org_settings (
  org_id uuid primary key references organizations(id) on delete cascade,
  settings jsonb not null default '{}'::jsonb,
  updated_at timestamptz not null default now()
);
alter table public.org_settings enable row level security;
-- read for any member, write for owner/manager only (mirror the finance-table pattern in the RLS model):
create policy "org read" on public.org_settings for select
  using (org_id in (select private.user_org_ids()));
create policy "org manage" on public.org_settings for all
  using (org_id in (select private.user_org_ids()) and private.user_role_in(org_id) in ('owner','manager'))
  with check (org_id in (select private.user_org_ids()) and private.user_role_in(org_id) in ('owner','manager'));

Expose via an OrgSettingsContext analogous to ExpenseCategoriesContext. Move LOW_STOCK_THRESHOLD, LICENSE_WARNING_DAYS, REPORTING_MONTHS and similar Dashboard magic numbers into this bag with the current values as defaults so behavior is unchanged until the operator edits them.

### Section 6 — Performance: Bundle, Assets, Client Render (Vite SPA)

_CE-OS is a Vite 6 / React 19 SPA with route-level code-splitting already in place (35 React.lazy routes in src/App.tsx), which is a real positive — recharts, qrcode, and per-page code are correctly kept out of the entry bundle, and @google/genai is confirmed server-only (NOT bundled into any client chunk). date-fns is used in exactly one place with a tree-shakeable named import. However the build still ships a 687 kB (203 kB gzip) main index chunk and a 328 kB (100 kB gzip) shared chunk, and two distinct heavy-dependency issues drive most of it. (1) The 328 kB "financeReports" chunk is actually the SHARED recharts vendor chunk: recharts (8.5 MB on disk) is imported eagerly by 6 lazy routes (Dashboard, FinancesOverview, Goals, Revenue, CashFlowReport, StatDetailModal), so Rollup hoists it into one shared chunk that loads on the first chart route. (2) framer-motion is dragged into the MAIN index chunk because CommandPalette (which imports framer-motion) is eagerly imported in App.tsx — confirmed by "framer"/"spring" markers present in index*.js. There is NO build.rollupOptions.manualChunks, so React/react-dom/supabase-js/framer-motion all sit in one undifferentiated 687 kB entry chunk that can never be cached separately from app code. The `motion` package is a verified DEAD dependency (zero imports anywhere in the repo). On the asset/render side: the Google Fonts Inter family is loaded via a render-blocking CSS @import in src/index.css:1 with no preconnect/preload; several raw <img> tags lack width/height/loading attributes; and the core data-fetching hook useEntity does a full `select("*")` table re-fetch (up to 50,000 rows) on every mount AND on every postgres_changes realtime event via an UNFILTERED subscription, with no caching — a render/network storm risk. ### Performance — Database & Query Layer

_The query layer has solid bones: finance aggregation is correctly pushed into Postgres SECURITY-INVOKER RPCs (finance_kpis/cashflow/alerts) so the heavy money math never pulls full tables into the browser, the RLS hot-path function private.user_org_ids() is STABLE and backed by an index on org_memberships(user_id), the .range() pagination loop correctly defeats PostgREST's silent 1000-row cap, writes are batched (updateMany/removeMany do one query + one setData), and the Vercel cron uses a stateless PostgREST HTTP call so there is no serverless connection-pool exhaustion risk. The systemic weaknesses are all on the client read path: (1) useEntity ALWAYS does .select('*') and loads the ENTIRE org-scoped table into memory on every mount — for expenses that is 1366 rows × 19 columns fetched on Vendors, VendorDetail, TaxReport and FinancesOverview pages that only consume 3 columns; (2) there is NO client cache (no React Query/SWR) so every navigation re-runs every fetch from scratch and the same table is fetched redundantly by multiple hooks mounted on one page (Dashboard mounts 5 full-table loaders, FinancesOverview loads orders+1366 expenses purely to back drill-down modals that may never open); (3) each useEntity/useOrders instance opens its OWN realtime channel that triggers a FULL table refetch on ANY org-wide change to that table, so N hooks on a table = N full refetches per write (amplification). On the DB side the live advisors are confirmed: 20 unindexed FKs, and — more impactful than the FKs given current row counts — the finance RPC filter columns are not covered by composite indexes (orders is filtered by (org_id, placed_at) but only org_id is indexed; expenses is filtered by (org_id, occurred_on) but the occurred_on index is keyed on user_id not org_id, so it can't serve the org-scoped finance path). At today's volumes (largest table 1366 rows) none of this is user-visible yet, but the over-fetch + no-cache + refetch-amplification triad scales linearly/multiplicatively with row count and is the thing to fix before the data grows._

**[F135] (High · effort L · conf high) No client-side query cache — every navigation refetches every table from scratch, and the same table is fetched redundantly by co-mounted hooks**  

- **Where:** `src/hooks/useEntity.tsx:65-100, src/hooks/useOrders.tsx:28-47, src/pages/Dashboard.tsx:40-44, src/pages/FinancesOverview.tsx:297-298`

- **Evidence:** There is no React Query/SWR (verified in context). useEntity holds state in local useState seeded empty on every mount, so leaving and returning to a page re-runs the full fetch. Dashboard.tsx:40-44 mounts useOrders + four useEntity hooks = five independent full-table fetches on every visit. FinancesOverview.tsx:297-298 additionally mounts useOrders (orders + nested order_items + customers join) and useEntity('expenses') (1366 rows) on top of its four finance RPCs — the orders+expenses pulls exist only to slice drill-down modals (openStat) that frequently are never opened. expenses is also independently re-fetched by Vendors, VendorDetail, TaxReport and FinancesOverview with zero sharing.

- **Fix:** Introduce a shared cache (TanStack Query is the smallest lift and integrates cleanly with the existing fetch shape) keyed by [table, orgId] with a sane staleTime so co-mounted consumers dedupe and navigation reuses cached data. Short of adding a dep, at minimum lazy-load the FinancesOverview drill-down data (orders/expenses) only when a stat modal opens rather than on page mount.


**[F136] (Medium · effort M · conf high) Realtime refetch amplification — each hook opens its own channel and does a FULL table refetch on any org-wide change to that table**  

- **Where:** `src/hooks/useEntity.tsx:107-121, src/hooks/useOrders.tsx:51-67`

- **Evidence:** Every useEntity/useOrders instance subscribes to postgres_changes on its table and, on ANY event, schedules `void fetchAll()` (a full re-pull of the whole org-scoped table) after a 250ms debounce. The subscription has no org filter (intentional, per the comment, because DELETE payloads only carry the PK), so a write by any user in the org triggers the refetch. With N hooks mounted on the same table (e.g. expenses is mounted by Vendors+VendorDetail or by FinancesOverview alongside its own), a single expense insert fans out to N independent full-table refetches. As tables grow, one teammate editing a row re-downloads the entire table for every open consumer.

- **Fix:** Centralize the subscription per [table, orgId] (a single shared channel feeding the shared cache) so one change = one refetch regardless of consumer count. With a cache in place, prefer applying the realtime payload incrementally (upsert/delete the changed row) instead of a blind full refetch; fall back to refetch only for DELETEs where only the PK is known.


**[F137] (Medium · effort S · conf medium) Finance RPC filter columns not covered by composite indexes — orders(org_id, placed_at) and expenses(org_id, occurred_on) fall back to org-only index + filter/sort**  

- **Where:** `DB indexes: orders_org_idx (org_id only), expenses_occurred_on_idx = (user_id, occurred_on), expenses_org_idx (org_id only); used by public._finance_kpi_window (orders WHERE org_id AND placed_at range; expenses WHERE org_id AND occurred_on range) and src/hooks/useOrders.tsx:35 (.order('placed_at'))`

- **Evidence:** _finance_kpi_window filters orders by org_id + (placed_at AT TIME ZONE)::date BETWEEN start/end and expenses by org_id + occurred_on BETWEEN start/end. The only orders index usable for that is orders_org_idx(org_id) — no (org_id, placed_at) composite — so Postgres reads all org orders then filters/sorts by placed_at. For expenses, expenses_occurred_on_idx is keyed on (user_id, occurred_on), but the finance path and RLS are org-scoped (no user_id predicate), so that index is NOT usable for the org window; it falls back to expenses_org_idx(org_id) then filters occurred_on. useOrders also sorts the full order set by placed_at with no supporting index.

- **Fix:** Add CREATE INDEX ON orders(org_id, placed_at) and CREATE INDEX ON expenses(org_id, occurred_on). These directly serve both the finance RPC range scans and the client list ordering. (Migration only — do not apply from here; flag for the user.) Note the placed_at predicate wraps the column in AT TIME ZONE/::date, so consider an expression index on ((placed_at AT TIME ZONE 'America/Phoenix')::date) if EXPLAIN shows the range predicate not using the plain composite.


**[F138] (Low · effort S · conf high) 20 unindexed foreign keys (confirmed live) — cheap to add, currently low-impact at present row counts**

**[F140] (Low · effort S · conf high) RLS auth_initplan: bare auth.uid() re-evaluated per row on profiles / access_requests / admin_emails policies**  

- **Where:** `pg_policies (public role): profiles 'profiles self select'/'profiles self update'/'profiles self upsert', access_requests 'admin select'/'admin update', admin_emails 'admin delete'/'admin select'`

- **Evidence:** These policies reference auth.uid() directly (not wrapped as (select auth.uid())), so Postgres re-evaluates the auth function per candidate row instead of once per statement (the auth_initplan advisor). The org-scoped data tables already route through private.user_org_ids() (STABLE, evaluated once) so they're fine; only these auth/admin tables have the bare pattern. Impact is small because profiles=2 rows, admin_emails=0, access_requests=3.

- **Fix:** Wrap as (select auth.uid()) in these policy predicates so it's evaluated once per statement. Cosmetic at current scale; worth doing for consistency with the rest of the schema. Migration only — flag for the user.


**[F141] (Nice-to-have · effort S · conf medium) profiles has two permissive SELECT policies — both OR-evaluated on every profiles read**  

- **Where:** `pg_policies public.profiles: 'profiles co-member read' + 'profiles self select' (both PERMISSIVE SELECT)`

- **Evidence:** Two permissive SELECT policies on profiles means Postgres evaluates the OR of both predicates for every row. profiles is only 2 rows and is read by useOrgMembers.tsx:48-49 and Settings.tsx:82-83, so cost is trivial, but multiple-permissive-policies is flagged by the advisor and adds a second per-row predicate.

- **Fix:** If 'co-member read' already subsumes self-visibility (a user is a co-member of their own org), collapse to a single policy; otherwise leave as-is — the row count makes this purely cosmetic.


**[F142] (Low · effort S · conf medium) _finance_kpi_window uses a per-order correlated lateral subquery for order_items quantity (mild N+1 inside the RPC)**  

- **Where:** `public._finance_kpi_window — fees CTE: left join lateral (select sum(qty) as items from public.order_items where order_id = o.id)`

- **Evidence:** Inside the channel-fee calculation, for each qualifying order the function runs a correlated subquery summing order_items.qty for that order_id. order_items_order_id_idx exists so each lookup is indexed, but it is still one index probe per order rather than a single grouped aggregate. At 224 orders / 289 items it's cheap; it grows linearly with order count and only matters if channel_fee_rules.listing_fee is set (the multiplier consuming `items`).

- **Fix:** Rewrite the lateral as a single `LEFT JOIN (SELECT order_id, sum(qty) items FROM order_items WHERE ... GROUP BY order_id)` pre-aggregated on the order window, so item quantities are summed once for the whole window instead of per order. Low priority until order volume climbs.


**[F143] (Low · effort S · conf high) PostgREST max-rows cap defeated by a serial page loop — correct, but pages sequentially (latency scales with table size)**  

- **Where:** `src/hooks/useEntity.tsx:71-87 (for-loop, await per 1000-row page)`

- **Evidence:** The DEFAULT_FETCH_LIMIT is 50000 and PAGE_SIZE 1000, looping until a short page ends the run. Pages are awaited serially (await inside the for-loop). For expenses (1366 rows) that's 2 sequential round-trips today; a table at 10k rows would be 10 sequential round-trips before the page can render, blocking on network latency. This is the correct way to beat the cap, but the combination with 'load the whole table on mount' (Finding 1) means total round-trips and payload both scale with row count.

- **Fix:** This is mostly mitigated by fixing Findings 1–2 (don't load whole tables; aggregate server-side). Where a full pull is genuinely needed, the serial loop is acceptable; if a table is expected to exceed a few thousand rows, prefer a server-side aggregate/paginated view over client-side full materialization.


**[F144] (Nice-to-have · effort S · conf high) Vercel cron correctly avoids a pooled DB connection (positive — no serverless pool exhaustion)**  

- **Where:** `api/cron/subscriptions.ts:32-37`

- **Evidence:** The daily cron issues a single stateless `fetch(${url}/rest/v1/rpc/process_due_subscriptions)` via PostgREST with the service-role key rather than opening a node-postgres/pg pool. This sidesteps the classic serverless problem of each cold invocation grabbing a direct Postgres connection and exhausting the pool. The DB advisor note about 'Auth DB connections absolute (not %)' is a Supabase auth-server config and unrelated to this app code.

- **Fix:** No action — documenting as a positive. If future serverless functions need direct SQL, route them through Supabase's transaction-mode pooler (port 6543), not a direct 5432 connection.
